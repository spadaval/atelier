Planning contract: root `atelier status` output.

Command name: `atelier status` at the root command group. It is a human-first signpost, not a machine JSON contract and not a replacement for focused drill-down commands.

Required sections: identity line with repository/workspace path; Local State with git cleanliness, branch/ahead-behind when available, and tracker/export/doctor summary; Active Work with current issue/worktree/session association or a no-active-work empty state; Active Mission with the single active mission ID/title/health and a compact progress count; Ready In Active Mission with a bounded list of immediately claimable issues; Blockers with the nearest blocking issue IDs and reasons; Recent Activity with the latest relevant issue/mission/evidence/activity entries; Next Actions with intent labels, for example inspect mission, start ready work, fix tracker health, inspect git state, or validate closeout.

Delegation boundaries: root status summarizes mission health only. Full linked-work inventory, evidence detail, and closeout validator output go to `atelier mission status <id>`. Full mission body and constraints go to `atelier mission show <id>`. Full activity streams go to history views. Issue detail goes to `atelier issue show <id>`.

Empty states: clean checkout still prints that local state is clean; no active mission gives mission list/create/add-work guidance; no active work points to ready active-mission work; no ready work distinguishes blocked, done, and no linked work; no blockers and no recent activity print explicit one-line empties; tracker/export failures print a short failure plus the focused command to inspect or repair.

Proof expectations: transcript tests cover clean and dirty git states, healthy and stale tracker/export states, no active work, active work, one active mission, no active mission, ready work, blocked work, no recent activity, and bounded output when many issues exist. Acceptance requires next actions to be intent-labeled instead of a generic command dump.

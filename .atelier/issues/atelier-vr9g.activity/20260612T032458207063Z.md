Issue transition options surface design.

Primary surface: add `atelier issue transition <id> --options` or an equivalent `atelier issue options <id>` command that answers what the issue can do next. The command should list allowed actions, blocked actions, fast gate reasons, and the command to perform each allowed action.

Relationship to existing commands:
- start action -> `atelier start <id>` as the preferred command, with `atelier work start <id>` as compatibility.
- finish action -> root `atelier finish <id>` if introduced, otherwise `atelier work finish <id>` remains the command.
- close action -> `atelier issue close <id> --reason "..."` when gates pass.
- reopen action -> `atelier issue update <id> --status open`; `issue reopen` becomes hidden compatibility if retained.
- status-only state changes -> `atelier issue update <id> --status <state>` for metadata/state edits, not proof-heavy transitions.
- mission closeout remains mission/status-led; issue transition options can mention mission blockers but should not substitute for mission closeout validation.

Validator relationship: `atelier workflow validate issue <id>` remains an advanced/compatibility guardrail and CI-like proof command. User-facing transition output should not require operators to know validator internals. Fast checks in transition options include open/closed status, blockers, parent/child constraints, active work, projection freshness, and required evidence links where those are already recorded. Expensive proof belongs in evidence records and validation transcripts, not synchronous transition calculation.

`issue show` expectation: summarize transition readiness in a compact section such as Next Actions, with one or two likely commands and blocked reasons. Do not dump every validator name or internal gate.

Help expectations: `issue transition --help` should say it lists allowed next states/actions and commands. `workflow validate --help` should position workflow validation as advanced policy checking used by closeout and automation, not as the normal discovery mechanism for user next steps.

Validation expectations: transcript coverage should include open ready issue, blocked issue, closed issue/reopen path, issue with active work, close-ready issue, missing-evidence close block, and consistency between `issue show` summary, `issue transition --options`, and `workflow validate issue` results.

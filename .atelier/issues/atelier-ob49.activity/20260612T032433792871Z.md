Command migration and removal policy.

Default behavior: moved commands get staged deprecation or compatibility aliases when removal would break current operator habits, documented Agent Factory guidance, or common transcripts. Direct removal is allowed only when the issue record explains why no compatibility path is needed and validation proves the replacement is clear.

Required contract for every moved or removed command:
- old command and new command mapping recorded in the owning issue or docs;
- primary help recommends only the new home;
- deprecated aliases are hidden from primary help or marked compatibility-only;
- normal alias output names the preferred replacement and performs the old action when safe;
- transcript coverage proves both replacement behavior and alias/deprecation behavior before removal;
- Agent Factory guidance is updated before removing commands it currently names.

Initial classifications:
- direct root aliases retained during migration: `export`, `rebuild`, `lint`, `doctor`, `diagnostics`, `work start`, `work finish`.
- hidden/staged aliases recommended: `issue quick`, `issue subissue`, `issue reopen`, `issue label`, `issue unlabel`, `issue blocked`, `issue block`, `issue unblock`, relation commands, `issue comment`, `issue search`, `issue tree`, `issue impact`, `import-beads`.
- removal candidates after replacement proof: `issue next`, `issue tested`, `issue close-all` unless a maintenance bulk-close workflow remains explicitly supported.

`issue quick` and `issue subissue`: primary help should remove them once `issue create --work` and `issue create --parent <id>` exist. Because current users may know those names, keep hidden deprecated aliases for one migration window unless transcript and docs show they were never part of supported guidance. Their alias output should be concise: created issue ID, canonical file path, and a preferred-command note.

Validation expectations: add transcript tests for preferred commands, deprecated aliases, removed-command errors, and help output. Removed commands should fail with a clear replacement message rather than a generic unknown-command error during the deprecation window. Documentation and Agent Factory guidance must never recommend removed commands.

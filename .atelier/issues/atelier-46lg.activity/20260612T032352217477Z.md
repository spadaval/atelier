Command taxonomy decision for setup, integrations, state, migration, and maintenance.

Target hierarchy:
- Setup/bootstrap: keep `atelier init` as the first-run command because it is already obvious and is useful before users know any group hierarchy. If a setup group is added later, `atelier setup init` may be an alias, but root `init` remains recommended.
- Integrations: keep optional external-tool setup under `atelier integrations ...`. Do not mix integrations with tracker state rebuild/export.
- State: introduce or document `atelier state export` and `atelier state rebuild` as the conceptual homes for projection and canonical-state synchronization. Existing root `atelier export` and `atelier rebuild` remain compatibility aliases because Agent Factory and current docs use them heavily.
- Migration/import: keep legacy transitions under migration, with `atelier migrate ...` as the group home. `atelier import-beads` should become a staged/hidden compatibility alias for a migration/import subcommand such as `atelier migrate import-beads` or `atelier migrate import beads`; direct removal waits for docs and transcript coverage.
- Maintenance/diagnostics: group record checks and local health under maintenance: lint, doctor, diagnostics, and any destructive maintenance such as issue deletion or bulk close if retained. Existing root `atelier lint`, `atelier doctor`, and `atelier diagnostics` remain compatibility aliases for at least one release because they are common closeout commands.

Validation expectations: top-level help must show the grouped mental model; group help must distinguish state rebuild/export from maintenance lint/doctor; aliases must print or document the preferred command; transcript coverage should include root aliases and preferred grouped commands for export, rebuild, lint, doctor, diagnostics, integrations, and migration/import. Agent Factory guidance can continue using root aliases until grouped commands are implemented and documented, then migrate deliberately.

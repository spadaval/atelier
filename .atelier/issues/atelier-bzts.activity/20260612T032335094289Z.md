Planning contract: `atelier prime` workflow command.

Decision: `atelier prime` belongs in the product as a recovery and onboarding command for agents, but it must not compete with `atelier status`. `status` answers the current checkout state. `prime` answers how an agent should operate in this repository and which command families matter. It is mostly static guidance blended with a small dynamic header that names the repo, active mission if one is discoverable, active work if one exists, and tracker health if cheap to compute.

Output sections: Context Recovery, including durable state location, ignored runtime/cache boundaries, and active mission/work summary; Core Rules, including use installed `atelier`, no JSON as the primary agent contract, canonical Markdown under `.atelier/`, and no predecessor tracker usage; Essential Commands, with one-line reasons for status, mission status/show, issue show/list-ready/update/close, work start/status/finish, evidence add/attach, workflow validate, export/check, lint, doctor, and rebuild; Common Workflows, including start assigned issue, recover after checkout/pull, validate and close an issue, mission closeout, and inspect history; Validation/Closeout Checklist, naming focused checks without implying every agent runs full tests; Repository Notes, loaded from repository binding or config when available.

Non-goals: it must not print long generic help, full command reference, implementation tutorials, or dynamic mission detail. It should point to help/docs for full command syntax and to `atelier status`/`atelier mission status` for live state.

Proof expectations: transcript examples compare useful `bd prime` concepts to Atelier-specific output, prove every listed command has a concrete reason, and show that `prime` remains useful in empty checkout, active mission, and active work states. Acceptance requires focused tests or snapshots for section order, empty dynamic header behavior, and no generic filler sections.

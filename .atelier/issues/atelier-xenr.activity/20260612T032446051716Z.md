File-first issue creation direction.

Primary creation path: `atelier issue create "Title"` allocates a valid issue ID, writes `.atelier/issues/<id>.md` with required front matter and a sparse editable body, refreshes projection, and prints the ID plus canonical Markdown path. The intended workflow is create shell, edit the Markdown file with normal file tools, then run `atelier lint <id>` or the focused validation command printed by create.

Options and old-to-new mapping:
- `atelier issue create "Title" --parent <id>` replaces `atelier issue subissue <parent-id> "Title"`.
- `atelier issue create "Title" --work` replaces `atelier issue quick "Title"` by creating the issue and associating or starting work according to the start/work policy.
- `atelier issue update --description ...` and other large text-field updates should be removed in favor of editing `.atelier/issues/<id>.md`; small metadata remains on `issue update`.

Alias/deprecation decision: primary help should show only `issue create` and its options. In tension with the original direct-removal preference, the mission-level migration policy now favors hidden staged aliases where habits could break. Therefore `issue quick` and `issue subissue` may remain hidden deprecated aliases for one migration window, but must not appear as normal creation paths. Direct removal is acceptable only after replacement transcripts, docs, and Agent Factory guidance show the file-first path is complete.

Output expectations: create output should include issue ID, status/type/priority defaults when useful, canonical file path, parent/work association summary when options are used, and next action such as `atelier lint <id>`. Quiet mode should remain script-friendly and emit only essential data.

Validation expectations: transcript tests should prove plain create, parented create, active-work create, canonical file path output, lint guidance, invalid parent failure, alias/deprecation output for quick/subissue if retained, and exported/rebuilt state consistency.

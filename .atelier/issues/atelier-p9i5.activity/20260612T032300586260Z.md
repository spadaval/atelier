Planning contract: shared work context model for signpost surfaces.

Define one internal WorkContext snapshot that status-like commands render in different slices rather than recomputing separate meanings. The snapshot should include: repository identity and path; git cleanliness/ahead-behind/conflict summary; tracker health and export freshness; current operator identity when available; active work association, including issue ID, title, status, parent, mission linkage, worktree path, and blocked/dirty flags; the single active mission identity and operational health; ready work limited first to the active mission graph; blocked work and named blockers; workflow validator gaps for the current target; evidence gaps; and recent activity from canonical issue/mission/evidence/activity records.

Rendering slices: root `atelier status` renders the checkout-level summary and immediate next actions; `atelier work status` renders active-work/worktree detail; no-ID `atelier mission status` renders the active mission if exactly one open active mission exists, otherwise an explicit empty/ambiguous state; `atelier mission status <id>` renders advancement and closeout detail for that mission; `atelier mission show <id>` renders the durable document record; issue/epic show commands may include compact parent/blocker/recent-activity slices but should drill down rather than duplicate mission status.

Empty states: no active mission means root status says there is no active mission and points to mission list/create/add-work; no active work means it points to ready mission work and `atelier work start <issue-id>`; no ready work means it distinguishes all work done, blocked work, and no linked work; no recent activity prints a one-line `No recent activity` rather than omitting the section.

Proof expectations: implementation is acceptable only when representative transcripts show the same active mission, active work, blockers, tracker health, recent activity, and next-action labels across root status, work status, mission status, and show/drill-down commands. Tests should cover no active mission, one active mission, ambiguous/multiple missions if policy allows that state, clean vs dirty worktree, ready vs blocked work, and empty recent activity.

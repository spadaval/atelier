Audit 2026-06-12: current agent-facing CLI surface and output pain points.

Scope sampled with installed atelier only, no JSON output: atelier --help; issue --help/list/list --ready/show/update --help/create --help/next/blocked; mission --help/show/status; work --help/status; worktree --help/status; evidence --help/list; workflow --help/validate issue/mission; diagnostics --help; doctor; attempted atelier status.

Findings:

1. Root command surface is categorized but not yet workflow-first. Evidence: atelier --help groups Setup, Issues, Missions and planning, Records, Work, State management, Integrations, Maintenance, plus common commands. This is clearer than a flat list, but the normal agent path is split across issue, mission, work, worktree, workflow, evidence, export, lint, and doctor, with no root status command. Problem: agents need several drill-down commands to answer "what should I do in this checkout?" Quality smell: high coupling between operator knowledge and command placement. Likely cause: inherited issue tracker commands plus newer mission/workflow concepts were added as peer groups. Value if fixed: agent confusion reduced and checkout readiness made visible from one signpost. Design implication: route atelier-rqvv/atelier-v02t/atelier-p9i5 around a root status surface that names active mission, active work, ready blockers, tracker health, and next useful drill-downs.

2. Issue command group is overloaded. Evidence: atelier issue --help exposes create, quick, subissue, list, search, show, update, close, close-all, reopen, delete, comment, label, unlabel, block, unblock, blocked, relate, unrelate, related, impact, next, tree, tested. Problem: lifecycle, graph editing, large text editing, queue/search, and compatibility helpers are mixed as equal choices. Quality smell: low cohesion and misplaced responsibility. Likely cause: legacy Chainlink issue surface absorbed agent-factory additions. Value if fixed: command choice becomes obvious and accidental destructive/editing paths are less prominent. Design implication: use existing blockers atelier-9jbu, atelier-51xk, atelier-xenr, atelier-vr9g, and atelier-o3w3 to split lifecycle essentials, file-first editing, transition options, and start/work relationship.

3. Next-command blocks are often syntactically valid but context-poor or stale. Evidence: issue show atelier-0qgv suggests issue comment, work start, and issue close even though work is already active and the issue blocks 30 follow-up items; mission show suggests mission add-work and workflow validate but does not explain closeout/readiness intent; issue next prints "Run: atelier session work atelier-0858", a removed command surface. Problem: footers encode command names rather than operator intent and can actively mislead. Quality smell: legacy drag and information leakage. Likely cause: per-command hardcoded strings instead of shared work-context/next-action formatting. Value if fixed: fewer invalid handoffs and less private context needed. Design implication: atelier-2qqk and atelier-ia6p should include intent labels, active-work awareness, and stale command scans/tests; atelier-ob49 should cover migration behavior for removed commands.

4. Status surfaces have overlapping partial truth. Evidence: work status correctly shows active issue, branch, worktree, and start time; mission status shows health blocked, closeout gates, active work, evidence gaps, and validators; mission show is durable record detail; atelier status is unrecognized. worktree status reports /root/atelier State: clean even while mission status/git status report tracker dirty entries from the current work association, and it emits a very large stale-export report for unrelated old worktrees. Problem: agents must reconcile multiple views to know checkout health. Quality smell: weak cohesion and change amplification across status renderers. Likely cause: work, mission, worktree, and git/export checks each own a slice of context. Value if fixed: clearer handoffs and safer closeout decisions. Design implication: atelier-p9i5 should define shared work context once; atelier-v02t should separate root status, mission operational status, and durable mission show; worktree status needs bounded/error-summary output or scoped drill-down.

5. Detail/list views vary between too much and too little output. Evidence: issue show atelier-0qgv prints all 30 blocking follow-up items plus recent activity; evidence list prints 40 long summaries; worktree status prints hundreds of stale missing-source lines for an unrelated worktree. In contrast, workflow validate issue/mission only reports durable_state_current pass by default for close transition, while mission status has the meaningful blockers/evidence gates. Problem: high-volume views bury the signal, while validator views omit context that agents expect when assessing readiness. Quality smell: weak test interface and information leakage. Likely cause: detail/list renderers are not consistently bounded and validator defaults are internal-policy centered. Value if fixed: transcript-based validation can assert scannable outputs and agents can choose correct drill-downs. Design implication: atelier-gtl6, atelier-o8a7, atelier-4x33, and atelier-vr9g should specify bounded lists, quiet-mode contracts, transcript fixtures, and whether workflow validate remains advanced while issue/mission transition views explain readiness.

6. Mutation and maintenance outputs are useful but uneven. Evidence: issue update --claim printed a projection refresh line before the actual update summary; doctor is concise and useful; issue create/update help still advertises large inline description/append-notes paths, while mission constraints call for human-readable CLI as primary interface and file-first durable records. Problem: acknowledgements sometimes mix repair/projection mechanics with user-facing change summaries. Quality smell: information leakage. Likely cause: projection refresh is surfaced from lower layers and mutation output is not standardized. Value if fixed: cleaner logs and more reliable quiet/script-friendly behavior without returning to JSON. Design implication: atelier-ve3w and atelier-o8a7 should define mutation summaries and quiet acknowledgements; atelier-c2oj should document file-first Markdown editing for large fields.

Recommended split next: keep existing follow-up graph rather than creating parallel items. Highest leverage order is (a) shared status/work-context contract: atelier-p9i5 plus atelier-v02t/atelier-rqvv; (b) next-action formatting and stale command cleanup: atelier-2qqk plus atelier-ia6p/atelier-ob49; (c) issue hierarchy reshaping: atelier-9jbu, atelier-51xk, atelier-xenr, atelier-o3w3, atelier-vr9g; (d) output style/quiet/transcript validation: atelier-gtl6, atelier-o8a7, atelier-4x33; (e) docs/Agent Factory guidance: atelier-dx73 after implementation decisions are made.

Risk: some verbosity is intentional for durable handoffs, and quiet mode already provides terse IDs for some reads. Before shaping implementation, each follow-up should preserve the human-output contract in docs/architecture/human-cli-output.md and avoid making full human output a scripting API.

Confidence: high for stale next command, missing root status, issue-group overload, and unbounded list/detail noise; medium for exact command reshaping, because compatibility and staged deprecation decisions remain open.
